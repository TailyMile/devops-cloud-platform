# Course 5 — Observability

## Goal

Introduce a production-style observability stack into the DevOps Cloud Platform.

The goal of this stage was to implement a complete monitoring and logging pipeline for a Kubernetes environment.

---

# Observability Architecture

The platform now includes two telemetry pipelines:

Metrics pipeline:

Kubernetes / node / workload exporters
→ Prometheus
→ Alertmanager
→ Grafana

Logs pipeline:

Pods
→ Alloy (DaemonSet log collector)
→ Loki
→ Grafana

---

# Components Introduced

## Prometheus

Prometheus is responsible for:

* collecting metrics from Kubernetes
* scraping node and workload metrics
* evaluating alert rules

Metrics sources include:

* kube-state-metrics
* node-exporter
* Kubernetes API
* application workloads

---

## Alertmanager

Alertmanager processes alerts generated by Prometheus.

Responsibilities:

* grouping alerts
* deduplicating alerts
* routing alerts to notification channels

Alerts are defined using Kubernetes `PrometheusRule` resources.

---

## Grafana

Grafana is used as the visualization layer.

It connects to:

* Prometheus (metrics)
* Loki (logs)

Capabilities:

* dashboards
* log exploration
* alert inspection

---

## Loki

Loki provides centralized log storage for the Kubernetes cluster.

Key characteristics:

* log indexing by labels
* efficient storage
* tight integration with Grafana

---

## Alloy

Alloy runs as a Kubernetes DaemonSet and collects logs from all cluster nodes.

Responsibilities:

* discovering Kubernetes pods
* reading container logs
* enriching logs with Kubernetes metadata
* forwarding logs to Loki

---

# Key Platform Capabilities Added

The platform now supports:

* centralized log aggregation
* cluster-wide metrics monitoring

# Implementation Approach

All observability components were deployed using Helm charts managed by Terraform:

* kube-prometheus-stack
* Loki
* Alloy

Helm values are stored in the repository and version-controlled.
